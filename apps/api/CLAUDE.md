# `@fantasytoken/api` — Backend Guide

Fastify v5 + PostgreSQL (Drizzle) + pino + TypeScript strict.

## Run

```sh
# from repo root
cp apps/api/.env.example apps/api/.env       # fill TELEGRAM_BOT_TOKEN
pnpm db:up                                   # start local Postgres in docker
pnpm --filter @fantasytoken/api dev          # tsx watch
```

## Layout

| Path | Purpose |
|---|---|
| `src/index.ts` | Bootstrap: config → logger → db → server → listen. |
| `src/server.ts` | Fastify factory. Registers plugins and modules. |
| `src/config.ts` | env validated via zod. **Fail-fast** on boot if invalid. |
| `src/logger.ts` | pino with PII redaction (INV-8). |
| `src/db/client.ts` | Drizzle client factory. |
| `src/db/schema/` | Drizzle table schemas. One file per domain. |
| `src/db/migrations/` | Generated by `pnpm db:generate`. |
| `src/lib/` | Cross-cutting helpers (auth HMAC, error class). |
| `src/modules/` | Domain modules. See `src/modules/CLAUDE.md`. |

## Critical invariants enforced here

- **INV-1** — `lib/telegram-auth.ts` validates initData on every authenticated request.
- **INV-7** — global error handler logs every caught error before responding.
- **INV-8** — pino `redact.paths` covers wallet addresses, tokens, initData. Extend when adding sensitive fields.

## Adding a new domain (checklist)

1. `mkdir src/modules/<domain>` — apply module convention.
2. `src/db/schema/<domain>.ts` — Drizzle table. Re-export from `schema/index.ts`.
3. `pnpm db:generate` — produces SQL migration.
4. Register routes in `src/server.ts`.
5. Add zod request/response schemas in `packages/shared/src/schemas/<domain>.ts` if they cross frontend/backend.
6. Write `<domain>.service.test.ts` first. Then implement.

## Maintenance triggers

- New Fastify plugin (cors, swagger, etc.) → register in `server.ts`, document above.
- Schema migration that's not backward-compatible → ADR with rollback plan.
- New env variable → add to `config.ts` schema AND `.env.example` AND root `.env.example` manifest.
