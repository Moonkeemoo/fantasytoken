Fantasy Token — MVP Wireframes

draft · for implementation

Функціональні low-fi варфрейми для 4 ключових екранів MVP: Lobby → Team Builder → Live Event → Result. Без візуалу, кольору й декору — лише структура, поведінка, дані й edge cases. Кожен екран = варфрейм + анотації по компонентах + state matrix + data spec + open questions. Візуальна оболонка — окремо (див. Design Research).

1Lobbybrowse · pick contest
▶
2Team Builderbuild lineup · pay entry
▶
3Live Eventwatch P/L · rank
▶
4Resultpayout · share · replay
Edge: contest sold out · payment fail · contest cancelled · 0 spots prize-won → see per-screen edge cases
01 · LOBBY
Browse contests
User scans available contests, filters free/paid, sees prize, entry fee, time left, spots filled. Picks one.
Success: tap a contest → opens Team Builder pre-loaded with that contest context.
02 · TEAM BUILDER
Build a lineup
Pick 5 tokens from a list, allocate 100% across them, optionally name the team. Confirm entry → debit balance / open top-up.
Success: entry confirmed → enters Live Event in waiting state until contest starts.
03 · LIVE EVENT
Watch performance
Live P/L, rank, time left, per-token contribution. Refresh every ~30 s. Read-only — no changes after lock.
Success: contest ends → auto-advance to Result.
04 · RESULT
Payout & share
Final P/L, final rank, prize won (or zero), per-token breakdown, share-card, "Play again" CTA.
Success: prize credited + user shares OR enters new contest.

screen 01 · lobby

Components

1
Header — identity + balance + top-up
Avatar, username, current cash balance, "+ Top up" CTA. Always visible. Tap balance → opens Wallet screen (out of MVP).
tap "+ Top up" → opens TopUpModal
2
Filter tabs
Three filters with counts: Cash (paid contests), Free (no entry fee), My (user's active + upcoming). Default: Cash.
tap → re-fetch list with filter
3
Featured contest card (hero)
One main contest, picked by backend (highest pool / nearest start). Shows: name, prize pool, entry fee, time-left countdown, spots-filled bar with "x/y" label, primary CTA.
tap card or CTA → open Team Builder with this contest pre-selected
4
Contest list
Scrollable list of remaining contests in selected filter. Each row: entry-fee badge, name, "Win up to $X · spots · time-left", JOIN button. Same tap target as hero card.
tap row → Team Builder; tap JOIN → Team Builder (same)
5
Active-contest banner (conditional)
Shown only if user has ≥1 contest currently live. Pinned above bottom nav. Tap → jump to Live Event for that contest (if multiple, opens picker).
visible only if liveContestCount > 0
6
Bottom nav (4 tabs)
Play (Lobby), Live (active contests list), Wallet, Me. MVP scope = first 2 functional; Wallet + Me can be stubs that link to a profile screen.
Play = current; Live = list of user's active contests

States

Default
≥ 1 contest available; user has balance ≥ 0; balance shown.
First-time
User has $0 balance. Show small banner above tabs: "Get $5 free credit" → onboarding flow.
No contests
Empty state: "No contests right now — check back in [next start time]" + CTA "Notify me".
Network fail
Skeleton loaders for hero + list. Retry button after 8 s timeout.
Contest sold out
List row shows JOIN disabled + "FULL" badge. Hero auto-replaces with next available.
Active contest
Banner #5 visible. If multiple active, banner says "X live now".

Data — what backend provides

GET
/contests?filter=cash|free|my
Returns
Array of Contest { id, name, entryFee, prizePool, spotsFilled, spotsTotal, startsAt, endsAt, status, isFeatured }
User
GET /me → { id, username, balance, activeContestIds[] }
Refresh
Poll every 30 s for spots-filled + time-left. Or single SSE stream.
Sort
Hero = isFeatured. List = startsAt ASC.

Edge cases

Tap full
Toast: "This contest is full" — list refreshes. Don't navigate.
Tap closed
If startsAt has passed and lock is on, route to Live Event (read-only) instead of Team Builder.
Cash tab, $0 balance
Contest rows show CTA as "Top up to join" instead of JOIN.

Open questions for product

  1. Чи потрібен пошук/фільтр по назві в MVP, чи 12-15 контестів читабельно скролом?
  2. Як обираємо "Featured" — алгоритм (next start + highest pool) чи редактор?
  3. Скільки контестів одночасно гравець може мати? Affects banner #5 logic.
  4. Чи показуємо "your team is in" індикатор поряд з контестом, де вже зайшов?
  5. Чи валідне правило "якщо нема балансу — показати Top up" чи краще "Free roll" як єдиний CTA?

screen 02 · team builder

Components

1
Context bar
Back arrow, contest name, entry fee, prize pool, time-left, step indicator (1/2 — picking, 2/2 — confirming if we split). Static during this screen.
back → returns to Lobby (warns if any picks made)
2
Lineup summary
5 slots. Each filled slot = token symbol + allocation %. Empty slots show dashed "+". Allocation bar below shows total %; turns green at exactly 100%.
tap empty slot → scrolls picker into focus; tap filled → opens edit/remove sheet
3
Token picker list
Available tokens (curated by backend per contest). Each row: symbol, name, 24h change, action. Action = + Add (auto-allocates equal share rebalance) OR stepper − % + when already added. Sort by 24h change default; toggle to alpha.
+ Add: rebalances all picks to equal split UP TO 5; stepper: ±5% per tap, never exceeds 100% total
4
Sticky confirm bar
Always-visible at bottom: entry fee, current balance, primary CTA. CTA disabled until lineup is valid (5 tokens AND total = 100%).
tap CTA → if balance < entryFee opens TopUpModal; else POST entry, debit, route to Live (waiting state)

Validation rules

Pick count
Exactly 5 tokens. Less → bar shows "X of 5". More → blocked at picker.
Allocation
Sum of allocations = 100%. Stepper increments are 5%. Min per token = 5%, max = 80%.
Balance
If balance < entryFee → CTA becomes "Top up $X to enter"; opens TopUpModal first, returns here on success.
Contest still open
Re-check on confirm; if startsAt passed during build → block + toast "Contest started — try another".

States

Empty
0 picks. CTA disabled, summary shows 5 dashed slots.
Building
1-4 picks. Summary fills, allocations stay valid via auto-rebalance.
Valid
5 picks, 100% allocated. CTA enabled, "✓ valid" indicator.
Over-allocated
If user manually adjusts and total != 100% → red bar + "needs Y%" copy. CTA disabled.
Insufficient balance
CTA copy: "Top up $X to enter". Tap → modal.
Contest closed mid-build
Show modal "Contest already started" + return to Lobby button.

Data

GET
/contests/:id → full contest + availableTokens[]
Token shape
Token { sym, name, change24h, priceUsd, iconUrl }
POST entry
POST /contests/:id/enter body: { picks: [{sym, alloc}] } → { entryId, lockedAt }
Returns
On success: redirect to /contests/:id/live?entry=:entryId
Errors
409 closed, 402 insufficient_balance, 400 invalid_lineup

Edge cases

Already entered
Backend returns 409 with existing entryId → route to Live with that entry. No double-entry in MVP.
Back during build
If picks > 0 → confirm sheet "Discard lineup?". Else navigate freely.
App closed mid-build
Persist draft locally (localStorage) for the active contest until contest startsAt. Re-open restores picks.
5th pick auto-fill
"Equal split" smart action when user adds 5th token: distributes remaining %.

Open questions for product

  1. Чи фіксована к-сть токенів = 5? Чи варіюється per contest (3–10)?
  2. Allocation step — 5% чи 1%? 5% простіше, але обмежує тонке налаштування.
  3. Min/max per token — чи треба обмежувати? 5–80% запропоновано, але можливо free-for-all?
  4. "Equal split" як дефолт — додавання токена авто-балансує до рівних %, чи лишає решту на ручне?
  5. Може гравець заходити кілька разів в один контест з різними lineup'ами (paid feature)? Якщо так — фактично тут break-glass для multi-team.
  6. Available tokens — куратор list per contest чи один глобальний на всі?
  7. Lock time — entry закривається з startsAt чи раніше (e.g. 5 min before)?

screen 03 · live event

Components

1
Header w/ live indicator
Back, contest name, time-left compact, big "● LIVE" badge. Time-left ticks every second client-side.
back → Lobby (no warning, this is read-only)
2
Main scoreboard
The hero. Big P/L number, original/current portfolio reference. Three sub-metrics: rank (delta vs last poll), projected prize, time-left.
refresh every 30 s; animate number tick on change
3
Lineup performance rows
5 rows = picks. Each shows: token, allocation, contribution bar (positive/negative), perf %, contribution $.
read-only in MVP; tap could open token-detail sheet (out of scope)
4
Mini leaderboard
Top 2 + user's row (highlighted). "VIEW ALL" → full leaderboard sheet (out of MVP scope OR included as simple modal).
tap "VIEW ALL" → leaderboard modal w/ all 247 rows

States

Waiting (pre-start)
If user enters before startsAt: scoreboard shows "Starts in MM:SS" instead of P/L. Lineup visible but no perf %.
Active
Live values updating every 30 s. Standard layout.
Stale data
If last poll > 90 s ago: subtle "Reconnecting…" pill near LIVE badge.
Connection lost
Banner: "Showing last known values · X min ago" + Retry.
Ended
On endsAt: auto-redirect to Result screen after 2 s celebratory state OR show "Final results coming up…" then redirect.
Cancelled by ops
Modal: "Contest cancelled — full refund issued" → return to Lobby.

Data

GET
/contests/:id/live?entryId=… every 30 s
Returns
{ portfolio: { startUsd, currentUsd, plPct }, rank, rankDelta, projectedPrize, lineup: [{sym, alloc, plPct, contribUsd}], leaderboardTop3, userRow }
Refresh
Polling 30 s. Future: SSE for real-time tick. Time-left ticks client-side off endsAt.
Auto-route
When endsAt ≤ now → fetch /result → push to /result screen.

Edge cases

Backgrounded
On foreground: force fresh poll within 1 s; if > 60 s gap show "catching up…" splash for max 3 s.
Tied rank
Display "T-#47" prefix when other entries share P/L exactly.
No projected prize
If user is below paying threshold: show "—" instead of $ value, with caption "top 30% pays".
Server time skew
Use server-provided endsAt always; never trust client clock for cutoff.

Open questions for product

  1. Polling vs SSE — для MVP polling 30s OK? чи на старті треба live-feel (5–10 s)?
  2. Projected prize math — реальний (за поточним rank+payout структурою) чи прихований до фіналу?
  3. Можна ділитись в Live (share P/L) чи тільки на Result екрані?
  4. Push-нотифікації в MVP? E.g. "rank changed", "5 min left", "you're paying".
  5. Full leaderboard — модалка чи окремий екран?
  6. Boost/skip — чи плануємо micro-actions (e.g. swap one token за rebuy fee)? Якщо так — frame це у фічу/non-MVP.

screen 04 · result

Components

1
Header — context
Close (×), contest name, "ended · X ago". "Final" tag clarifies snapshot.
× → Lobby
2
Headline result
Big number = NET (prize − entry). Caption: final P/L %, final rank "/of N". Share + "View on chain" actions.
Share → opens native share sheet w/ pre-rendered share-card image + URL; View on chain → opens block explorer for prize tx (out of MVP if no on-chain payout)
3
Breakdown
Hairline list: entry fee (−), prize won (+), net (highlighted). Plain math.
none — read-only
4
Lineup recap
5 rows showing each token's final P/L %. Same layout as Live, but without contribution bar — just final number.
none in MVP
5
Bottom CTAs
Two buttons: "Lobby" (ghost) and "Play again" (primary). "Play again" → routes to Lobby with the same contest type filter pre-selected, OR if the same contest has a "next session" → directly to Team Builder for the next one.
Lobby → / ; Play again → /lobby?prefill=:filter or directly to next session

Result variants

Won (in money)
NET > 0. Headline: "you won $X". Confetti / celebration moment (small, 1 s).
Won prize, net loss
Won < entry fee — rare but possible (e.g. tiered tail prize). Headline: "you won $X · net −$Y". No celebration.
No prize
Below cutoff. Headline: "no prize this time". Show "you were close" copy if rank ≤ cutoff + 10. Highlight free roll suggestion.
Cancelled
Headline: "contest cancelled · refund $X". Same layout but no rank/perf. Single CTA: "Back to Lobby".
First-place
Special hero state: "you won 🏆 · $X" + bigger share emphasis. (Spec for visual layer; functionally same.)

Data

GET
/contests/:id/result?entryId=…
Returns
{ outcome: 'won'|'no_prize'|'cancelled', prizeUsd, entryFeeUsd, netUsd, finalPlPct, finalRank, totalEntries, lineupFinal: [{sym, alloc, finalPlPct}], shareCardUrl, txHash }
Idempotent
Stable result — once written, won't change. Safe to cache aggressively.
Share-card
Pre-rendered PNG generated server-side at result time, URL returned. Used in Share intent.

Edge cases

Direct nav before end
If user opens /result before endsAt → redirect to /live.
Refund pending
If cancelled and refund not yet posted: show "refund processing · X min" + retry mechanism.
No share card
If shareCardUrl missing → fall back to native share with text only.
Replay context
"Play again" must read which filter (cash/free) the original contest belonged to. If contest type no longer exists → land on default Lobby.

Open questions for product

  1. Виплата — миттєвий кредит на in-app balance чи on-chain TX? Affects "View on chain".
  2. Share-card — генеруємо server-side (recommended) чи client-side? Який brand?
  3. "Play again" — чи відкриваємо одразу новий contest (next session) чи завжди Lobby?
  4. Free roll mechanic — якщо програв, пропонувати free contest? І чи це affects retention?
  5. Breakdown depth — показувати fees / rake explicit? Affects user trust.
  6. Result archive — де гравець бачить історію? Profile screen (out of MVP)?
  7. Top-3 окремий extra — спецдиплом / NFT / badge для перших трьох? (visual only OR mechanic?)

What's NOT in this spec (out of MVP): auth/onboarding, top-up modal flow, KYC, full leaderboard screen, profile/history, tutorial, settings, push notifications, multi-team, social feed, in-app messaging, chain integration UX. Each can be specced separately when scope allows.

Reading order for engineering: 1) Components (what to build), 2) Data (contracts), 3) States (what to render when), 4) Edge cases (what can break), 5) Open questions (need answers BEFORE coding).
