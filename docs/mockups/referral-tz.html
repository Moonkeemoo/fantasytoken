§ 01 · Overview & goals

Why we're building this

Drive viral acquisition through a paid referral program. Users earn a percentage of their friends' contest winnings — forever, while the friend keeps playing. Both sides get a one-time cash bonus after the friend's first contest.

Success metrics

  • Activation — % of new users who arrive via referral link (target: 30%+ at steady state)
  • Conversion — % of referred users who complete first contest within 7 days (target: 50%+)
  • Retention multiplier — D30 retention of referred vs organic (expect 1.4×)
  • K-factor — avg invites sent per active user (target: 0.4 at month 3)

Out of scope (this milestone)

  • Tier-based commission rates (e.g. higher % for Gold tier referrers)
  • Group/team referrals
  • Anti-fraud KYC flow — handled separately by trust & safety

§ 02 · Reward mechanics

Two reward types, two levels

One-time bonus

RecipientAmountTriggerLocked?
Referee (new user)$25On first contest completion (not just join)Yes — until trigger
Referrer (existing)$25Same — credited to balance immediatelyNo

Lifetime commission

LevelRateSourceApplies to
L1 direct5%Friend's contest winnings onlyUSD, STARS, TON
L2 indirect1%Friend-of-friend's winningsUSD, STARS, TON
Critical: commission applies to net winnings (prize − entry fee), not gross prize pool. Commission is paid from house margin, never from the friend's payout. Friends never see less because they were referred.

Per-currency rates (override)

For STARS and TON, rates differ slightly to absorb on-chain costs:

CurrencyL1L2
USD5%1%
STARS3%0.5%
TON2.5%0.5%

§ 03 · Data model

Core tables

referrals

-- one row per referrer→referee link
id              uuid PK
referrer_id     uuid FK users
referee_id      uuid FK users (UNIQUE — user can only be referred once)
level           smallint  -- always 1; L2 derived via JOIN
code_used       text      -- e.g. 'REF-TARAS'
source          text      -- 'tg-share' | 'qr' | 'copy-link' | 'channel'
created_at      timestamptz
activated_at    timestamptz NULL  -- set when referee completes 1st contest

referral_codes

user_id         uuid PK FK users
code            text UNIQUE  -- 'REF-{handle uppercased}', fallback 'REF-{base32(8)}'
created_at      timestamptz

referral_commissions

-- one row per commission event
id              uuid PK
referrer_id     uuid FK
referee_id     uuid FK
contest_id      uuid FK
level           smallint  -- 1 or 2
currency        text      -- 'USD' | 'STARS' | 'TON'
winnings        numeric   -- friend's net winnings
rate            numeric   -- 0.05, 0.01, etc
amount          numeric   -- credited to referrer
created_at      timestamptz

referral_bonuses

-- one-time $25 bonuses
id              uuid PK
user_id         uuid FK   -- recipient (referrer OR referee)
role            text      -- 'referrer' | 'referee'
referral_id     uuid FK referrals
amount          numeric DEFAULT 25
currency        text DEFAULT 'USD'
status          text      -- 'pending' | 'credited' | 'expired'
expires_at      timestamptz  -- referee bonus: 7d after signup
credited_at     timestamptz NULL

§ 05 · Attribution & tracking

Attribution rules (in order)

  1. If user opens app via ?startapp=ref_X → attribute to code X.
  2. If user already has a Telegram account that's signed up → no attribution (one-shot).
  3. If code is invalid/self-referral → log warning, proceed without attribution.
  4. Attribution window: 7 days from first link click → first signup.

Bonus crediting

  • On signup: create referrals row, two referral_bonuses rows (status=pending).
  • On referee's first contest completion (any outcome): mark both bonuses credited, push to balance.
  • If referee doesn't complete a contest within 7d: expire referee bonus, leave referrer bonus pending until 30d.

§ 07 · SURFACE 2

Lobby · promo carousel

Invite is one slot in a rotating promo strip

Lobby gets a horizontal carousel of "promo" cards above the Open Contests list. Featured contest is the primary slot (always shown first). Other slots — Next unlock, Invite friends — rotate behind it.

Position
Lobby · directly under user header bar · above "Open contests" list
Layout
320×168 cards · 10px gap · 1 active visible + peek of next · horizontal swipe
Indicator
Dots above carousel · active dot stretches to pill (16×5px)
Auto-advance
None on first load · auto-advance only after manual swipe (respect user choice)

Slot priority logic

  1. Featured contest — always slot 1, sourced from /lobby/featured
  2. Invite friends — slot 2 if user has <5 active referrals · otherwise slot 3
  3. Next unlock — slot 2 if user is <1000 XP from next tier · otherwise slot 3
  4. If user has 5+ referrals AND is not close to next tier → drop those slots, show only Featured + system promos

Invite slot — content

  • Kind label: INVITE FRIENDS · F-tile mark in top-right
  • Title: "Earn 5% forever"
  • Sub: "Both get $25 after their first contest"
  • Meta: "Per friend" / "$25 + 5%"
  • CTA: "Send link →" (accent button)

§ 09 · SURFACE 4

Telegram bot DM · hourly digest

Batched commission notifications via TG bot

The bot DMs the user when their referrals win contests. Batched hourly to avoid notification fatigue. Sent via standard sendMessage + InlineKeyboardMarkup.

Frequency
Max 1 message per hour · only if commission events occurred in window
Quiet hours
22:00–08:00 in user's TG timezone · accumulate, send next active window
User control
"Pause hourly digest 🔕" inline button → opt-out (still tracks commission, no DMs)
API
sendMessage with parse_mode=HTML · inline keyboard with web_app button to open Mini App

Message content

  • Opening line: "{N} friends just won contests · last hour"
  • Indented block (rendered with HTML <pre> or formatted text):
    • Per friend: avatar initial · name · contest · level pill (L1/L2) · commission amount
    • Footer row: "Total this hour" · sum in green
  • Inline keyboard:
    • Row 1: [🎯 Open app] (web_app)
    • Row 2: [See breakdown] · [Invite more]
    • Row 3: [🔕 Pause hourly digest]
Note on TG capabilities: TG bots cannot render arbitrary HTML/CSS in messages. The "data table" inside the message is plain text with monospace alignment (<pre> or manual padding). Rich card visuals require sendPhoto with a server-rendered PNG — acceptable fallback if monospace looks weak in client tests.

Singleton case

If only 1 friend won in the hour: simplify copy to "Andriy just won $87 in Bear Trap. You earned +$4.35." Same inline keyboard.

§ 12 · Edge cases & abuse prevention

CaseBehavior
Self-referral (user uses own code)Reject at attribution · log self_referral_attempt
Already referredReject · keep original attribution · show toast "Welcome back!"
Code expired/invalidProceed signup without attribution · do NOT show error to user
Referee never completes contestBoth bonuses expire · referrer notified once at 30d "Andriy hasn't played yet — give them a nudge?"
Referee makes contest, then refundsReverse commissions · debit referrer balance (with note in tx log)
Friend deletes accountMark referral inactive · stop commission accrual · keep historical records
Suspected fraud (multi-account)Freeze bonuses, escalate to T&S · don't auto-deny (false positives)
Currency mismatch (referee plays in TON, referrer in USD)Commission accrues in friend's currency · referrer wallet shows multi-currency balance

Hard limits

  • Max 100 active referrals per user (soft cap, contact support to lift)
  • Max $50,000 lifetime commission per user before manual review
  • Same IP signup within 24h → flag, don't block

§ 15 · Don't lose the referrer link

The referee → referrer relationship is permanent and visible everywhere it matters

Once attribution happens, we treat referrals.referrer_id as immutable, indexed, and surfaced. Engineering must not silently drop it during signup retries, account migrations, or wallet linking. Below — where it lives in DB, when it's locked, and where the user actually sees it.

Database guarantees

  • referrals.referee_id is UNIQUE — one referrer per user, forever. Re-attribution is impossible.
  • referrals.referrer_id is NOT NULL on the referee side — never write a referral row without both ends.
  • Index idx_referrals_referrer on referrer_id (we query "all referees of X" constantly).
  • Index idx_referrals_referee on referee_id (we query "who referred X" on every contest finish).
  • Cascade rules: if referrer deletes account → mark referrer_status='inactive', do not delete the row. Commissions stop, history stays.

Locking the link

  • Attribution is write-once. Once referrals row exists for a referee_id, no API path can overwrite it.
  • Account merge / wallet relink → carry referrer_id to the surviving account. Migration script must explicitly preserve it (add to merge test suite).
  • If a TG account is reset (same tg_user_id, new internal user_id) → carry the link via tg_user_id lookup.

Where the referrer is shown to the referee

SurfaceWhat's shown
Welcome screenReferrer's name (in note yellow) + avatar in hero. Persistent — always rendered if attributed.
Profile · /meSmall footer line under track record: "Invited by Andriy · Jan 12" with avatar tap → opens referrer's public profile.
First contest resultIf referee just unlocked the $25 bonus → toast: "Bonus credited · thanks to Andriy".

Where the referee is shown to the referrer

  • Profile expanded view — full friends list with avatar, name, status (active / pending / inactive), L1 / L2 pill, lifetime contribution.
  • Bot hourly digest — names of friends who won that hour.
  • In-app toast — name of friend whose win triggered the commission.
  • 30d nudge — if referee never played: "Andriy hasn't played yet — give them a nudge?" with deeplink to TG chat.

Edge cases that must NOT drop the link

ScenarioRequired behavior
Signup fails midway, retriedIdempotency key on POST /referrals/attribute · same start_param → same referrals row, never duplicated, never dropped.
User reinstalls TG Mini AppLink persists — keyed by tg_user_id, not session.
Welcome screen skipped / closedAttribution already happened pre-Welcome · screen is informational only · skip ≠ unlink.
Referee changes handle / display nameReferrer's friends list reflects new name on next refresh · link unchanged.
Referrer changes handleOld code_used string in referrals stays for audit · current code regenerates on demand.
Bonus expires (referee never played)Bonus row → status='expired' · referral row stays · if referee plays later → no bonus, but commission still flows.
Hard rule for engineering: any code path that touches users, auth, or wallet tables must be reviewed against "does this preserve referrals.referrer_id?" Add a regression test for every such path. Losing the link silently = lost commission = customer support hell.

Monitoring

  • Dashboard metric: referrals_orphaned (rows where referrer_id no longer exists in users) — should always be 0.
  • Dashboard metric: signups_with_ref_param_no_attribution — attribution failures · alert if > 1% of signups.
  • Weekly audit: count of referrals created vs ref_attributed events — should match exactly.
